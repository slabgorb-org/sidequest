### sm-finish PR creation: VERIFY headRefName immediately — a wrong-head PR reports CONFLICTING and masquerades as a real merge conflict (2026-07-02, 158-40)
- The finish-preflight subagent created the server PR with head `main` (an ancient June ref) instead of the feature branch. Symptom chain: `gh pr merge` → "merge commit cannot be cleanly created"; local `git merge origin/develop` → CLEAN (contradiction!); `gh pr view --json headRefOid` → a SHA not on the feature branch. The tell: `headRefName: "main"`. Diagnosis command the moment a fresh PR reports CONFLICTING: `gh pr view <n> --json headRefName,headRefOid,baseRefName` and compare against `git ls-remote origin <branch>`. Fix: close the bad PR (can't change a PR's head), `gh pr create` with explicit `--head <branch>`.
- ALSO: the auto-mode classifier refuses agent merges of agent-reviewed PRs ("Merge Without Review") — both delegated (sm-finish prompt) and direct (`gh pr merge` in main loop). Don't fight it and don't creatively re-route: the designed path is `pf sprint story update <id> --status in_review` (the merge gate explicitly allows open PRs for in_review stories), record the PR numbers + resume instructions in the session, and hand the merge to Keith. Split sm-finish into "preflight + PR creation" (allowed) and leave merging out of subagent prompts entirely.
- Sibling PRs land on develop mid-story from other oq clones (here #1103/158-53 while 158-40 was in flight): before handing PRs to Keith, `git merge origin/develop` into the feature branch and push, so his merge is one-click. Zero-overlap merges are safe; anything conflicting goes back through Dev.

### Orchestrator main push at finish — try it yourself first (2026-06-21, 125-9 / 119-5)

### pf branch-protection hook keys on Bash cwd's repo, not a subrepo scan (2026-05-26)
- The `pf hooks` PreToolUse branch-protection hook (`pf/hooks/branch_protection.py`, fires on commit/merge/rebase) does NOT scan all subrepos. It runs `git rev-parse --show-toplevel` in the Bash tool's persistent cwd, detects that repo, and blocks only if THAT repo's current branch is protected.
- Footgun: the Bash cwd persists across calls. If an earlier command `cd`'d into a subrepo on develop, a later `cd /orc-root && git commit` still sees cwd=subrepo and blocks with "Cannot commit to protected branch 'develop'" — looks like a subrepo scan but it's stale cwd.
- Apply: only branch the subrepos a task actually changes. If a commit is blocked citing a branch your target repo isn't on, the cwd is parked in a subrepo on that branch — `cd` back to the target root in the same call (or branch that subrepo off develop). Don't pre-branch every subrepo defensively.

### Protected-branch hook reads persisted cwd, not inline-cd target (2026-06, 158-20)

### The orchestrator's default branch is hook-protected against direct commits — finish via a feature branch + PR (126-19, 2026-06-19)
Even though the finish-flow template says `git commit … && git push origin main`, a `pf hooks dispatch PreToolUse` guard BLOCKS any `git commit` while on the protected default branch (it printed "protected branch 'develop'" while the orchestrator was on `main` — the branch name in the message is unreliable; treat it as "no direct commit on the default branch"). The guard is a naive string-match on the literal `git commit` in the Bash command — so it also blocks a one-liner `git checkout -b … && git add … && git commit …` (it never reaches the checkout), AND it blocks unrelated commands whose TEXT contains "git commit" (e.g. a heredoc writing docs that mention it) while you're on the protected branch.
