### encountergen enemy blocks carry NO creature_id — name is the only join back to the bestiary; and a new `pack.<method>` read breaks SimpleNamespace loader-stubs (2026-06-20, 157-2)
- **The bestiary join key is the NAME.** `encountergen.generate_enemy_from_bestiary` (`sidequest/cli/encountergen/encountergen.py`) builds the EnemyBlock with `class_="creature"` (a literal) and `name=entry.name` (verbatim) — it does NOT carry `entry.id`/`creature_id`. So any code that needs to map a seeded encounter's enemies back to their `BestiaryEntry` (e.g. epic-157's `pregen._encounter_factions` unioning faction tags) MUST join by `name` (case-insensitive), not id. `MonsterManual.find_enemy_by_name` already does this for the same reason. If you ever need a robust id join, you'd have to add `creature_id=entry.id` to the EnemyBlock first.
- **Adding a `pack.effective_X()` call breaks the test loader-stubs.** `pregen.seed_manual` does `pack = load_genre_pack(dir)`; tests monkeypatch `load_genre_pack` to a `SimpleNamespace` (`test_pregen.py::_stub_pack`) that hand-mimics ONLY the pack methods seed_manual reads (`effective_cultures`, `effective_archetypes`). Add a new read (`pack.effective_bestiary(world)`) and every stub-based test `AttributeError`s. Fix = add the method to the stub (`pack.effective_bestiary = lambda _world: (None, "stub")`) — a real `GenrePack` always has it, so this is matching the stub to the real type, NOT weakening production. Don't reach for a getattr-default in production to paper over the stub (that's a silent fallback; a real pack never lacks the method, and `pack is not None` already guards load failure).
- **The server suite has a KNOWN pre-existing failure baseline (~88–96, flaky by xdist/infra) — see the baseline note below.** When verifying a change, an ADR-145 "genre catalog must be mode=verbatim" inventory-validation failure at *pack load* (e.g. `test_pregen_fail_loud_90_5`, `test_encountergen_bestiary_90_1`, items in `tests/fixtures/packs/test_genre/inventory.yaml`) is part of that baseline, NOT your regression — confirm by (a) `git status` showing your diff touches no inventory/validator/fixture file and (b) `git log -1 --format=%ci -- <validator> <fixture>` predating your branch base. A throwaway `git worktree` of develop can't run server tests (the venv lacks `claude_agent_sdk`, provisioned out-of-band in the main `.venv`), so causal-trace + timestamps is the practical proof.

### De-nativizing a Fate confrontation SEAT: the marker is `win_condition`, NOT `enc.contest` (2026-06-20, 126-30)
- Story 126-30 (Keith ruling 2026-06-19): under `ruleset=='fate'` a standoff/conflict must seat as a pure Fate conflict (4dF + ablative stress → taken-out), with the native `opponent_metric.tension` dial REMOVED — the upstream half of #964 (#964 only gated the native overlay PAYLOAD off via `should_emit_native_confrontation`; seating in `instantiate_encounter_from_trigger` still stamped `win_condition='dial_threshold'` + tension metrics).
- **The trap:** TEA's HOW-agnostic RED accepts EITHER `enc.contest is not None` OR `win_condition != 'dial_threshold'`. Do NOT reach for `enc.contest` — `dispatch_fate_action`/`dispatch_fate_defense` (`server/dispatch/fate_conflict.py`) select the Contest engine (first-to-N victories) vs the Conflict engine (stress→taken-out) on **`encounter.contest is not None`**. A stress-based conflict MUST keep `contest=None`, so stamping a Contest would route a standoff to the WRONG engine. The de-nativization marker therefore has to move `win_condition` off `dial_threshold`.

Separately gate the WHOLE native combat-seeding block on `not is_fate` (`if cdef.category=="combat" and not is_fate:`) — TEA's Improvement: it leaks `_seed_combat_hp_depletion_to_npcs`/`_publish_combat_edge_to_npcs` AND `_roll_and_persist_initiative` CRASHES under Fate (FateConfig has no DEXTERITY map). The opponent FateSheet (the real win track) is already seeded by `_seed_fate_opponents` (runs for every Fate confrontation).
- **`win_condition` Literal vs `WinCondition` enum are DELIBERATELY decoupled.** `StructuredEncounter.win_condition` is a `Literal[...]` string (NOT the `genre.models.rules.WinCondition` enum) to avoid a `game`→`genre.models` import cycle (see the field docstring). So an ENGINE-ONLY runtime value like `fate_conflict` — which content never authors — goes in the **Literal only**, never the content enum. Safe because every native reader of `enc.win_condition` is an open `== "dial_threshold"`/`== "hp_depletion"` check (no exhaustive switch exists — audit with `grep 'enc\.win_condition\|encounter\.win_condition'`), and `dial_threshold_outcome()` already returns None for any non-`dial_threshold` value. The native overlay readers (`build_confrontation_payload`, `beat_filter`) are gated off for Fate anyway.

A non-Fate pack change here is a behavioral NO-OP (`is_fate=False` → identical control flow), so any SWN/WWN-suite diff is flakiness — confirm by running it `-n0` in isolation (the `test_firefight…hp_depletion` "regression" passed 3/3 solo).

### NarratorDirectives must render through ONE player-safe boundary — raw `[{kind}]` tokens leak machinery into player prose (2026-06-19, playtest must_not_narrate leak)
- Symptom (annees_folles playtest): the narrator printed its own machinery to the PLAYER — *"The must_not_narrate constraint is clear … I cannot narrate Roy picking up … an envelope that doesn't exist."* Root cause was a ONE-LINE render in `orchestrator.py` (group-C directive block): `block = "\n".join(f"- [{d.kind}] {d.payload}" …)`. The raw `NarratorDirectiveKind` token (`must_not_narrate`, `must_narrate`, `distinctive_detail_for_referent`, `canonical_only_do_not_reveal_to_others`) went into the prompt as a visible label with NO framing that these are silent stage directions → the model parroted the token AND broke frame to explain the constraint.
- Fix pattern: a single render boundary `sidequest/agents/narrator_directives.py::render_narrator_directives(directives)` maps each kind → an in-fiction imperative carrying NO machinery vocabulary ("Keep out of the fiction", "Bring into the scene", …), wrapped in `<stage-directions>` framing that says obey silently + show absence IN-WORLD (hand finds only what's there) rather than break frame. Import-time `assert set(map)==set(get_args(NarratorDirectiveKind))` fails loud if a new kind is added unmapped (No Silent Fallbacks).
- Reusable lessons: (1) any enum/token that reaches an LLM prompt as a label is a leak risk — the model will echo it; route ALL narrator-facing machinery through one mapping that emits prose, never the token. (2) `get_args(SomeLiteral)` + a module-load completeness assert is the clean fail-loud guard for "every kind must be handled". (3) Tests that assert `"must_not_narrate" in prompt` were codifying the LEAK — three prompt tests (`test_lethality_directives_in_prompt`, `test_group_c_e2e`, `test_group_c_wiring`) asserted the raw token; the fix flips them to assert the imperative + keeps the payload assertions. The strict no-token regression lives on the pure render fn, not the whole-prompt (avoids fragility if another section legitimately names the concept).
- Baseline caveat (reconfirmed): oq-1 `develop` carries ~92 pre-existing full-suite failures (WN combat / epic-108 in-flight, chargen-e2e, dice/spell-cast, content). Prove a change is clean by DIFFING the failure SET before/after (stash -u + re-run), not by the count — and re-run a lone "new" failure with `-n0` to rule out xdist flakiness (one did exactly this here).

### When wrapping an `async for x in query(...)` loop to map raised transport errors to a typed error, re-raise the already-typed error FIRST (119-4 green, 2026-06-16)
- Pattern for mapping a raised SDK/transport exception (absent/expired login surfaces as a raised `query()` exception, OQ-5) to a typed domain error without double-wrapping: order the handlers `except AgentSdkAuthUnavailable: raise` BEFORE `except Exception as exc: ...raise AgentSdkAuthUnavailable(...) from exc`. The broad handler must use `from exc` (preserve the cause) and emit the GM-panel event BEFORE the raise (No Silent Fallbacks — the failure must be *visible*, not just loud). Re-indenting the loop body under the new `try:` is an unavoidable but mechanical diff; ruff-format it after and the only real change is the wrapper.

### A stored-but-unread field (and any docstring/comment claiming behavior the code doesn't do) is a HONESTY rejection even when the feature works green end-to-end (2026-06-16, 118-5 green-rework)
- 118-5 was functionally correct, fully wired, Fate suite green — and Reviewer REJECTED it. Not a functional rejection: a **quality/honesty** one. AC-1 said "store the offered fate point delta", so Dev added `PendingCompel.offered_delta` (default 1) — but `fate_projection` dropped it and the UI button hardcoded `Accept (+1 FP)`. The field was **stored but never projected or read** (No Stubbing / dead-code violation) and its docstring claimed "the fate point the player GAINS by accepting" — a lie, because nothing read it. A hardcoded display literal shadowing a stored field is also a **divergence vector** the OTEL lie-detector doctrine forbids (engine could grant X while UI shows a stale literal — the El Dorado lie).
- **Rule — when an AC says "store X", wire X end-to-end: store → project → display.** The displayed value must READ the real datum (`+{c.offered_delta}`), never a hardcoded literal that can drift. If a field is genuinely forward-only/unused, that contradicts the AC — escalate; don't leave it dead. Adding the field is half the work; reading it is the other half.
- **Reconcile honesty with bind-don't-balance:** keep the engine economy SRD-fixed (accept = +1 via `earn_fate_point`; don't make the grant read a variable field — that converts an SRD constant into a homebrew tunable, ADR-144). Instead PIN the field's default to the SRD value at its single creation site (`add_pending_compel` exposes no override) so display / stored field / grant agree **by construction** and the field cannot lie. The display reads the field (anti-drift); the engine stays SRD.
- **Wiring-test docstrings must claim ONLY the path the test renders.** `GameBoard-fate-compel-wiring.test.tsx` rendered `<GameBoard>` directly (injecting `onFateAction` as a prop) yet its docstring claimed it proved the `App → GameBoard` production path (`App.tsx:2727 → GameBoard.tsx:644`). It never mounts `<App>`, so that link is untested — comment-analyzer flags this HIGH and the reviewer **cannot dismiss** it (same as the 107-1 precedent). Trim the claim to the GameBoard→Surface path actually exercised and name the App→GameBoard seam as the remaining untested link (or mount `<App>`). Also delete stale "FAIL today / (RED)" lines once the test is green.
- **Negative tests on shared fail-loud paths:** even when accept and refuse share one `find_pending_compel is None → raise`, the reviewer wants BOTH legs tested independently (phantom-accept AND phantom-refuse) so a future routing divergence can't slip a never-offered verb through. Same for guard branches (`not encounter.resolved`): pin the resolved branch (span fires, nothing persists) so the guard can't be silently deleted.

### A scary `git diff develop..branch` (huge churn, file MOVES) does NOT predict a painful rebase — the conflict set is the INTERSECTION of branch-touched and develop-touched files (2026-06-16, 114-12 reconciliation)
- A stale 114-12 branch showed `+1491/-3278` across 68 files vs develop, including `{server => foundation}` honest-layering moves — looked like a nightmare rebase. But the branch's REAL changeset (`merge-base..branch`) was only 5 files, and `comm -12` of (branch-touched-files) ∩ (develop-touched-files-since-merge-base) was **empty**. The rebase replayed with **0 conflicts**. The giant diff was all develop's drift on files the branch never touched.
- **Rule before any scary rebase:** compute `MB=$(git merge-base develop $BR)`, then `comm -12 <(git diff --name-only $MB..$BR|sort) <(git diff --name-only $MB..develop|sort)`. Empty → conflict-free rebase regardless of net-diff size. After rebasing, the authoritative no-regression check is `git diff --name-only develop..HEAD` — it must equal exactly your intended changeset; anything extra means you clobbered develop's work.
- **Direction matters on reparents:** ADR-147 honest-layering moves pure logic DOWN (`sidequest/server/* → sidequest/foundation/*`), not up. Don't assume the direction — `git ls-tree -r develop --name-only | grep <file>` to see where develop actually keeps it before calling a file "resurrected."

### A full-suite "N failures" after a rebase/checkout can be PRE-EXISTING develop debt — verify on untouched develop before attributing it to your change (2026-06-16, 114-12)

### A test that reads `worlds/<world>/inventory.yaml` directly sees the WORLD tier ONLY — the genre-tier verbatim items it merges with are invisible, and dict `.get(id, {})` hides the drop (2026-06-15, 120-5)

### Reconciling a COMPETING already-merged implementation: conflict markers are only half the job — the competitor's net-new files ride in SILENTLY (2026-06-13, 106-1 dup-impl)
- Story 106-1 was implemented twice in parallel; develop already had a competing impl (#836) that this branch had to SUPERSEDE per Keith's ruling. `git merge origin/develop` flagged the two overlapping files (`chargen_loadout.py`, `chargen_mixin.py`) as `UU` conflicts — easy to spot. Those are **additions on `theirs` only → no conflict marker → they merge in clean and silently**, leaving orphaned dead spans + a test asserting the OLD span vocabulary that would either fail or rot.
- **Rule:** after resolving the `UU` files, enumerate what the competitor PR added that you're replacing: `git show <their-sha> --stat`. For each net-new file/section the loser added, decide explicitly. If your branch never touched that file and only the loser modified it on develop (`git log <base>..origin/develop -- <file>` shows only their commit AND `git diff <base> HEAD -- <file>` is empty), then `git checkout HEAD -- <file>` restores your (base) version, cleanly dropping their addition. For loser-only new files, `git rm -f <file>`. Then grep the whole tree for the loser's symbol names to confirm zero orphans before committing the merge.
- **Watch for accidental-agreement collisions:** both impls independently defined `SPAN_CHARGEN_ARMOR_EQUIPPED = "chargen.armor_equipped"` (mine in `inventory.py`, theirs in `chargen.py`). Restoring base `chargen.py` left exactly one definition — but if I'd taken theirs I'd have had a duplicate constant + double `SPAN_ROUTES` registration. Always grep the span-name STRING, not just the Python constant name.

### Don't trust the briefing's "server runs --reload / PID X" — verify `ps -o lstart` against the commit time (2026-06-13, beneath_sunden playtest)
- Briefing said "oq-1 uvicorn IS running --reload" and a later pingpong note claimed a FIXER bounce gave fresh PIDs. ### pf protected-branch hook reads the branch BEFORE your compound command runs — `git checkout -b X && git commit` still blocks (2026-06-12)

### Git discipline
- **`cd` persists within a single Bash call.** `cd foo && git -C foo …` resolves to `foo/foo`; the shell resets only *between* calls. Use absolute paths with `git -C` throughout (never mixing a `cd`), or re-anchor with `cd <subdir>` at the start of each chain.
- **Hard ban on `git stash`** — no exceptions, including read-only diagnostics and post-merge "safe" cases. Stashes poof; branches are durable. Set work aside via `git switch -c tmp-<reason>` + commit; to peek upstream use `git show origin/<branch>:<path>`.
- **Never stash during an active merge.** It strips MERGE_HEAD; `stash pop` injects fresh `<<<<<<<` markers on partly-resolved files and the first commit becomes a one-parent commit instead of the two-parent merge GitHub needs to mark the PR mergeable (45-3 incident). Inspect mid-conflict with read-only `git show origin/<branch>:path`.
- **Never stash + checkout develop to check if a failure is "pre-existing."** Wastes time, risks the tree, irrelevant — pre-existing or not, the job is a green suite (Keith, 2026-05-25).
- **Subrepos are independent git repos**, normally on `develop`; a `git commit` inside one lands on *that* subrepo's branch, not the orchestrator feature branch. Per gitflow each subrepo needs a `feat/*` branch. Before dispatching any implementer that touches a subrepo, check `git branch --show-current` and create the matching `feat/<feature>` branch first (2026-05-10: six subagents committed straight to develop).
- **Stacked-branch commits can silently orphan.** Commits that logically belong on parent B but were made on child C don't survive B's squash-merge, then `git rebase --onto develop B C` drops them via patch-id dedup (2026-05-09, 12 commits lost; recovered via `git reflog --all` cherry-pick). Before squash-merging a stacked parent, verify its tip with `git log --stat parent..child -- <files>`.
- **oq-1 and oq-2 are separate orchestrator clones**, each with its OWN subrepos that can drift across branches. Always dispatch subagents / verify with ABSOLUTE paths (`git -C /abs/path`), never bare `cd sidequest-server`. Mass test failures at pack-load are a top symptom of content-clone branch drift — check `git -C .../sidequest-content branch --show-current` FIRST (`main` lacks beneath_sunden/Epic-24 YAML). After `gh pr merge --delete-branch`, gh reverts the tree to the stale local default branch.
- **macOS BSD `sed` silently no-ops `\b`** — use literal patterns + grep-verify. Make a safety branch before any non-trivial rebase.

### Postgres substrate (ADR-115) traps
- **Secondary writes inside an open turn transaction MUST ride that tx, or they self-deadlock under PG (2026-05-26, D8).** `emit_event` opens `with repo.transaction() as tx:` which takes a per-session `FOR UPDATE` row lock on `sessions` (connection A). Any telemetry/side-write reached *inside* that block (e.g. the projection fan-out's visibility-gated `_publish_secret_routed` → `watcher_hub.publish_event` → `PgTelemetrySink.record`) that opens its OWN pooled connection takes a SECOND `session_tx` `FOR UPDATE` on the same locked row → self-deadlock. **Harmless on SQLite** (one in-process connection, no row lock) so it only surfaces under PG on a real MP turn. Fix: thread `tx`/`event_seq` (optional, default `None`) down the call chain so the inner write rides the same connection; lazy-fill / out-of-frame callers keep `None` and safely take their own short `session_tx` (no row lock held). ## To stop a transport-error handler from swallowing your OWN bugs, drive the async iterator by hand — wrap only `anext()` (119-4 green rework, 2026-06-16)

119-4's narrator loop was `async for message in query(...)` inside one broad
`except Exception` that mapped anything raised to `AgentSdkAuthUnavailable` + a
`narrator.auth_unavailable` event. That catch couldn't tell "the SDK transport
failed (= absent login, OQ-5 → really is auth)" from "our message-processing
raised (a parse `IndexError`)" — both are inside the `async for`, so a parser bug
lit the GM panel red for "login expired." The Reviewer killed it as a dishonest
signal on the lie-detector panel.

**The fix is structural, not a type-narrow.** A `except ClaudeSDKError` narrow
fails the AC1' tests, whose fakes raise a bare `RuntimeError` to mean "the
transport boundary raised." Instead, drive the iterator manually so the catch
wraps ONLY the boundary:

```python
stream = aiter(query(prompt=prompt, options=options))
while True:
    try:
        message = await anext(stream)        # transport boundary ONLY
    except StopAsyncIteration:
        break
    except Exception as exc:                  # transport failed → map to auth
        emit_auth_event(reason="query_raised", ...)
        raise AgentSdkAuthUnavailable("... login absent/expired OR a transport "
                                      "error ...") from exc
    # --- OUR processing, OUTSIDE the catch: its bugs propagate RAW ---
    process(message)                          # IndexError here is NEVER auth
```

- `aiter()`/`anext()` are builtins (≥3.10); the server runs 3.14. An async
  generator is its own iterator, so `aiter(query(...))` then `anext(...)` works
  for the real SDK and the `FakeQuery`/`RaisingFakeQuery` seams alike.
- The boundary catch still maps a transport raise to auth (AC1' wants the typed,
  panel-visible signal), but the HEADLINE must name the alternative ("...or a
  transport error") — never assert a diagnosis the boundary can't prove.
- Verify the wrapping context manager **re-raises** (doesn't suppress): our
  `llm_request_span` does `except Exception: record; raise`, so an internal error
  escapes cleanly. Check that before relying on propagation.
- Regression matters: the restructure touches the hot narrator path. Full
  `tests/agents/` (1975 tests) stayed green — run the whole blast radius, not just
  the story file, after an iteration-mechanics change.
