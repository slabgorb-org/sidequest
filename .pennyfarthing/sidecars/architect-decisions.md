### ADR hygiene: current accuracy beats historical preservation (2026-05-02, from Keith)
- When auditing ADRs, optimize for giving CURRENT agents an accurate view of what is ACTUALLY running. Historical documents matter much less. (Keith's clarification during the ADR-057 audit.)
- Top-of-document deprecation callouts beat buried implementation-status sections when a design is dead — make stale ADRs LOUD about being stale so future agents see ground truth in three seconds.
- Don't write retrospective "successor" ADRs that just say "we kept the default" — that adds reading load with nothing beyond what code already documents. Point at code as source of truth liberally (`narrator.py:80` outranks an ADR that abstracts it).
- ADRs stay immutable history (preserve files, don't delete) but annotate aggressively when reality diverged. Bias to `deprecated`/`historical` over successor-ADR ceremony when the architecture was abandoned by inaction, not replaced.
- Successor ADRs are still correct when a real new design replaced an old one (ADR-091 over 043, ADR-092 over 069) — wrong only when there's just a never-replaced default.

### A port is a port, not a redesign — translate faithfully, don't brainstorm (2026-06-05, 87-2 / 86-3)
- When work is framed as a port of an already-working system (Rust era, sq-2, EH/WWN binding), treat it as translation: don't invoke `superpowers:brainstorming`, don't run an options/alternatives menu, don't reach for "modern" Python alternatives, don't open design questions on parts the user hasn't said are open. ADR-082/087 are the explicit port decisions. Output a translation table (source file → dest, key transforms), apply only the deltas the user names.
- Content/ruleset ports follow the same rule: "replace, don't remap." Keith on heavy_metal 5e→WWN: "I don't care one bit about what is there now, don't bother 'remapping', just replace with the port." Don't field-by-field translate the old content or preserve its flavor — author the faithful WWN port and replace the scaffolding wholesale (genre tier AND the live world's chargen path).
- For a Without-Number binding the existing native-dial confrontation is dead prose being REPLACED — do not reconcile/layer/preserve it, and never present "wire-up-existing vs faithful-port" as a fork (87-2, 86-3 CWN chase). NEVER ask "how faithful" / "how much to reskin" / offer a small-starter-set / defer part of it — the answer is always: faithfully port the whole source system, verbatim mechanics, flavor reskin at most. Asking the fidelity question on a port is the exact failure this prevents.

### ADR frontmatter schema invariants the pre-commit hook enforces (2026-05-17, from Keith)
- The orchestrator pre-commit hook (`pf hooks schema-validation`) validates staged `docs/adr/*.md` frontmatter and BLOCKS the commit on violations. Two non-obvious invariants:
- (1) `status: superseded` ⇒ `implementation-status: retired`. A superseded ADR may NOT carry `partial`/`live`/`drift`. The "what remnant survives on a legacy/opt-in path" nuance goes in a BODY banner, never the frontmatter enum (`live|partial|drift|deferred|not-applicable|retired`).
- (2) Supersession must be SYMMETRIC. If ADR-X says `supersedes: [Y]`, then ADR-Y must say `superseded-by: X`. An umbrella/migration ADR that spawned dedicated successors must NOT list the originals in `supersedes` if those originals point at the successors — list only the ADR whose `superseded-by` points back, keep the rest in `related`. (ADR-101 supersedes [1] only; 39/58/28 → successors 102/103/104.)
- How to apply: set superseded ADRs to `retired` up front, nuance in a loud body banner; before adding to `supersedes`, check the target's `superseded-by` points back; regenerate indexes (`scripts/regenerate_adr_indexes.py`) after edits. Root-cause fix, never `--no-verify`.

### Fate (4dF) player rolls must be physics-is-the-roll, not server-RNG-then-animate (2026-06-17, from Keith)
- Keith, emphatically: the dice must be DETERMINATIVE. The player THROWS the dice and the settled faces ARE the roll (physics-is-the-roll, the ADR-074 model the d20/DICE_RESULT path already uses). It does NOT mean server rolls RNG then animates dice to match.
- The Fate path was built backwards: `resolve_action -> roll_4df(rng)` rolls server-side and `FateDiceTray` 3D dice are post-hoc decoration → dice tumble to physics-determined faces UNRELATED to the server RNG and can contradict the text readout. Story 125-4 doubled down (synthesized an arbitrary throw gesture so dice "animate") — the failure.
- Apply: for any player-facing dice surface, the throw IS the roll — never animate over a pre-decided server result. "Mirror DICE_RESULT" = mirror the mechanism (client throws → reports settled faces → server resolves from them; spectators replay throw_params+seed → same faces by construction), not just the message shape. Server-side `roll_4df` is correct ONLY for actors with no client to throw (NPCs/opponents). Fix filed as story 126-7. 125-4's merged `throw_params`/`seed` wire fields + dF `'0'` label are correct groundwork, NOT reverted.

### Anti-pattern rules to enforce architecturally
- **No keyword / pattern matching for intent classification** (ADR-010, ADR-032). The Zork Problem. Natural language defeats finite verb sets every time. Intent is always an LLM call (preferably folded into the narrator's Opus response for zero additional latency). Reject any subsystem design that adds keyword heuristics to intent routing, dispatch, or narrator-output interpretation.
- **No text-synthesis dispatch for structured UI actions.** UI button clicks must send dedicated protocol messages (`BEAT_SELECTION`, `TACTICAL_ACTION`, `JOURNAL_REQUEST`, `CHARACTER_CREATION`). Never synthesize natural-language `PLAYER_ACTION` strings from structured UI state. Reference: the 2026-04-11 `05a3dfb` incident where beat-button clicks were fuzzy-matched against narrator label text.
- **No AI judging AI.** Second-LLM validators for narrator output, consistency checks, or game-decision verification are the "God lifting rocks" problem. The first call is already the most semantic reader available.
- **No live LLM calls in test suites.** `cargo test` must mock `ClaudeClient`. Live-LLM suites live in `--ignored`.
