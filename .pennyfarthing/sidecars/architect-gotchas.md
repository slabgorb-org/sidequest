## Architect Gotchas

### Subagent-driven-development on this repo — three traps (2026-06-22, dungeon-region-population)
- **Never put `ruff format .` (repo-wide) in a plan's final gate step.** `develop` carries format drift, so `ruff format .` reformats 30-40 UNRELATED files into the working tree (formatting-only, but pollutes the PR). Scope it: `uv run ruff format <feature files>` or `uv run ruff check --select I` for import order. One SDD run left 42 uncommitted reformats (36 non-feature) from this single plan line.
- **Implementer/fixer subagent dispatches MUST carry the "never `git stash`; use temp branches" rule.** A general-purpose subagent used `git stash` to test a baseline (forbidden per git-gotchas). They don't inherit the sidecar — put the constraint in the dispatch prompt for any task that might compare against develop.
- **The PreToolUse hook blocks a compound Bash command containing `git commit` whenever the CURRENT branch is protected (`develop`/`main`), even if the same command runs `git checkout -b feat/...` first.** It evaluates the branch at dispatch time. Split: create the branch in one call, then `git add`+`commit` in the next.
- **`uv run pytest` here runs under xdist (`-n auto`); some tests are flaky under parallelism** (e.g. `test_watcher_events.py::test_publish_event_shape` — WatcherHub builtins-pinned singleton, ADR-132, cross-test contamination; PASSES in isolation). Confirm a suspected regression by running it ISOLATED before blaming the diff. Also baseline-verify "pre-existing" failure claims by checking the failing file is outside `git diff --name-only <merge-base>..HEAD` — do NOT trust a subagent's `git stash`-based "pre-existing" proof.

### The procedural dungeon's curate stage is DEAD OUTPUT — generated creatures are computed then discarded (2026-06-22, 153 epic)
- `materializer.py` `_stage_curate` (~1404-1530) builds `RegionCuration.region_creatures[region_id]` + `region_big_bad[region_id]` for EVERY procedural region. The accessors `creatures_for_region()`/`big_bad_for_region()` (`materializer.py:512-524`) have **ZERO non-test callers** (grep it). Nothing persists or injects them. They evaporate when `materialize()` returns.
- The room-YAML emit (`room_yaml_emit.py:66-72`; `_stage_emit_room_yamls` `materializer.py:2116`) writes only `{room_type,name,description,entities}` — it NEVER writes `encounter_creatures`. But the runtime binding resolver `room_creature_binding.py:resolve_room_creatures` READS `encounter_creatures`. **The generator never writes the key the runtime reads**, so procedural rooms (`exp00N.rN`) field an empty creature pool and the narrator improvises monsters with no engine backing (fabricated HP, invented rules → Lore). Only HAND-AUTHORED room YAMLs (e.g. entrance's `gnaw_swarm`) ever surface a creature.
- `CuratedCreature` is free-text (name/type/telegraph/HP-from-CR, `materializer.py:965-1019`) — NOT a bestiary entry, so it CANNOT ride the authored `encounter_creatures`→bestiary→`_creature_patch_from_bestiary_entry` path. Wiring it = direct NPC-patch inject keyed by region_id (like `_creature_patch_from_enemy` does for encountergen rows), persisted per-save via the existing `record_mutation(region_id, kind=...)`/`load_mutations()` freeze seam (NO new table). Design: `docs/superpowers/specs/2026-06-22-dungeon-region-population-wiring-design.md`.
- **Co-location seating keys off the NARRATOR-owned free-text scene, not the region id.** `encounter_lifecycle.py:948/1023/1127` compares `npc.last_seen_location`/`npc.location` vs `party_location(pc)` (free-text). The creature's `location` is stamped from PRE-dispatch `turn_context.current_location` while its room binding uses POST-dispatch `region_for()` (`websocket_session_handler.py:838-862`) — on a movement turn they diverge → creature carries the prior room's name → `NoOpponentAvailableError` → confrontation never seats. Fix = region-aware co-location (`npc.region == region_for(pc)`), self-gated by an `npc.region` stamp so the 11 non-procedural worlds are untouched.

### The intent router does NOT thrash — it runs once/turn, bounded; the latency villain is the narrator loop (2026-06-17, 126-9)
- Recurring false hypothesis on this project: "the intent router is thrashing" causes latency. It is not. `IntentRouter.decompose` runs EXACTLY once per player turn, one Haiku call (×2 only on hard failure, never on low confidence), fenced by `INTENT_ROUTER_CALL_BUDGET_PER_TURN=2`; the dispatch bank runs once, no retry. `phase_call_counts.intent_router_pass = 1` across 339k `turn_telemetry` rows. The 8×/turn storm that earned the router its bad name was dice-replay re-running it — already FIXED by `suppress_intent_router`.
- When turn latency regresses, look at `agent_duration_ms` (the narrator SDK tool-loop: up to 8 heavy-model round-trips, 16 on dice turns) FIRST, not the router. The Timing tab (`#/dashboard` tab 4, story 124-3) settles router-vs-narrator immediately and is the lie detector for this whole class of question.
- Fate caveat (separate, MEDIUM-confidence lever — NOT thrash): `intent_router_pass.py:364-365` injects `build_fate_projection` (PC skills + ALL live aspects) into BOTH the narrator and router prompts when `pack.rules.ruleset=='fate'`. On bloated Fate prompts the single structured router call intermittently can't finalize inside the mandatory `max_turns=2` floor → `error_max_turns` fragility, surfacing as slow single calls (37-81s observed on annees_folles). This enlarges the prompt; it does not loop. Trimming `build_fate_projection` is the lever — tracked separately from the narrator thinking fix.

### Fate / ruleset dispatch — beats vs FATE_ACTIONs
- **Fate confrontations DO NOT traverse the beat/`apply_beat` path.** A `ruleset: fate` pack resolves through `dispatch_fate_action` (FATE_ACTION: overcome/create_advantage/attack), not the legacy dial/`beat_selection` dispatch in `narration_apply.py`. The genre `beats:` on a Fate confrontation are author-facing flavor that maps to the four SRD actions; the engine never runs `apply_beat` for them. When designing any Fate mechanic, the trigger lives on the FATE_ACTION path, not the beat path. (Found 2026-06-17 designing the spaghetti_western Standoff→Gunfight escalation — the spec's "draw beat fires it" mechanism was un-implementable for exactly this reason.)
- **`BeatDef` is `model_config = {"extra": "forbid"}` (`genre/models/rules.py`).** You cannot author a new per-beat flag (`escalates: true`, etc.) — it fails pack load. Escalation/lifecycle signals belong at the confrontation level: `ConfrontationDef.escalates_to` already exists and is the reuse hook. Reach for the existing reject/branch points (e.g. attack-in-Contest already raises in `fate_conflict.py`) before inventing a new payload or schema field.
- **Fate has no d8 initiative.** The Conflict engine (`run_fate_exchange`) walks Notice/Empathy *skill* order; the `encounter.initiative` list is SWN/WN-only. "Acts first" in a Fate scene = a boost/free-invoke edge, NOT an initiative entry. Don't design Fate ordering against the WN initiative model.

### The wiring failure class (the dominant bug category in this project)
- **Don't reinvent — wire up what exists.** Before building anything new, grep the codebase. Many systems are fully implemented but not wired into the server or UI. CLAUDE.md rule #3.
- **No half-wired features.** Connect the full pipeline or don't start. If something needs 5 connections, make 5 connections. Shipping 3 and calling it done is the exact failure mode Epic 15 was created to clean up.
- **Wire it, don't just define it.** Adding fields to a struct is not wiring. Adding props to a component is not wiring. Data must actually flow from source → consumer. If no code populates and transmits the field, the feature isn't done.
- **Never rationalize unwired code.** When `/sq-wire-it check` reports FAIL (no non-test consumers), the answer is **wire it**. Do not write paragraphs explaining why this time is different. If I catch myself typing "this is acceptable because" — STOP. That sentence is the bug. The wiring usually takes 5 minutes; the rationalization takes longer.
- **"Wiring means dashboard."** Internal data flow without OTEL visibility is not wired. A missing `WatcherEventBuilder::new(...)` call on a subsystem's dispatch path is a blocking wiring bug, never a "non-blocking improvement." Labeling an OTEL gap as non-blocking in my own audit is the exact self-compounded failure I hit in the 2026-04-09 Map fix.
- **No agent rationalization.** Dev/Architect/TEA/Reviewer are all Claude — they don't get to split blame across personas. "The Architect said it was fine" is not a defense. If a wiring gap is identified in any phase, the only valid response is "fix the code now," never "clarify the spec" or "file a follow-up story."
- **Verify wiring before claiming done.** `grep` for non-test consumers on the merged branch before marking a fix as complete. Don't wait for Keith to ask. A playtest with the GM panel open is the acceptance test.
- **Verify end-to-end before claiming fixed.** Code review is not verification. Trace the full wire path: source → channel → writer task → WebSocket → client handler → DOM. Session 7 had six consecutive "fixed" claims on the turn-lock bug, each found a real issue, none verified the message actually arrived at the browser.
- **Wiring gaps require action, not acknowledgment.** Finding a wiring gap and saying "yep that's a wiring gap" is worthless. Fix it. Log it. Check for siblings (if one compute-then-ignore exists, grep for the pattern). Verify OTEL. Don't move on.
- **Kitchen-sink gate failure is real** (story 16-1): nine agents approved unwired code because every phase rationalized it as "scope boundary" and "wiring comes in subsequent stories." The wiring-check gate exists in `.pennyfarthing/gates/` — every agent must enforce it, not just the reviewer as a backstop.

### No stubs, no fallbacks
- **No stubs. Ever.** When I hit a scope error or type mismatch, solve the actual problem. Don't substitute `Default::default()`, placeholder values, or restructure the design to avoid the issue. CLAUDE.md line 124: "Never say 'the right fix is X' and then do Y." Do X. If `snapshot` isn't in scope, move the code to where it is.
- **No silent fallbacks.** If a path/config/resource isn't where it should be, fail loudly. No `if not exists: try_other_thing`, no `unwrap_or_default()` that papers over missing data, no `Option::None` degradation on required fields.
- **No layered fallback design.** If the LLM is down, the system is down. Keyword intent classification "working" while narration is broken is not meaningful resilience — it's dead code with delusions of usefulness.
- **No reading `~/.pennyfarthing`.** Stay within the project directory tree. Global pennyfarthing config pulls in work-project settings that don't apply to SideQuest.

### Build / test gotchas
- **Build verification happens on OQ-2, not OQ-1.** All edits live in OQ-1/sidequest-api; after merge, pull on OQ-2 and `cargo build -p sidequest-server` there. The workspace-root build is a placeholder.
- **Test compile cascade:** Never spawn two cargo processes on the same workspace at once. Cargo's build lock queues them and a 2-minute timeout + new spawn creates zombie-compile cascades (4 competing cargos, 10+ minutes, zero results). Use `timeout: 300000` (5 min), `cargo build` first, `cargo test` second.
- **0.02s cargo build means nothing changed.** If the build is suspiciously fast, your fix isn't in the binary. Look for "Compiling sidequest-server" in the output as proof the change landed.
- **Server logs exist and are informative.** Check `/tmp/sq-api.log` before speculating about causes.

### spec-check: new payload fields must clear the extra="forbid" protocol boundary (73-4)
- When a story adds a key to a `build_*_payload()` dict, the GREEN tests can be fully green while the feature still crashes in production. The protocol models (`sidequest/protocol/messages.py`, e.g. `ConfrontationPayload`) are `extra="forbid"`, and the broadcast site wraps the dict: `ConfrontationPayload(**payload_dict)` (websocket_session_handler.py + make_confrontation_frame_supplier). An **undeclared** key raises `ValidationError` every emit.
- Why tests miss it: payload-builder wiring tests assert on the raw dict (pre-wrap); UI tests use the TS wire type which is `[key: string]: unknown` (permissive). Neither exercises the pydantic boundary.
- Spec-check move: for any new payload key, grep the model — is the field DECLARED? (`win_condition`/`player_hp` are declared; that's why they work.) If not, it's a Critical Option-B handback. The closing test is `ProtocolModel(**build_*_payload(...))` round-trip.
- UI boundary is usually safe: `App.tsx` casts the whole wire payload to the component data type (`msg.payload as unknown as ConfrontationData`) — no field-picking — so additive fields pass through automatically. Verify this rather than assume it.

### Legends have TWO authoring forms; ~all real worlds use the directory form (2026-06-03, story 65-12, spec-check)
- **What happened:** The 65-12 timeline loaded legends via `_load_legends_flexible(world_dir/"legends.yaml")` (flat-file form). Green tests passed (18/18) because every fixture used the flat `legends:` map form. Spec-check caught that `genre/loader.py:1037-1048` loads legends from EITHER a per-file `worlds/<slug>/legends/` DIRECTORY (the `legends_dir.is_dir()` branch, `Legend.model_validate` per file) OR the flat file (`_load_legends_flexible`, else). A scan of live content showed **~19 of the live worlds use the DIRECTORY form and have NO flat legends.yaml** (five_points, evropi, franchise_nations, annees_folles, aureate_span, glenross, oz/wonderland/gulliver, the_circuit, …). So the timeline silently rendered nothing for nearly all real worlds — an AC failure masked by fixtures that matched the implementation's blind spot.
- **Why it matters:** When a reference/projection feature reads a content file that the genre loader also reads, it MUST handle every authoring form the loader accepts — not just the one your fixtures use. "Tests green" means nothing if the fixtures and the implementation share the same blind spot (the canonical TEA failure). For legends specifically: directory form is the DOMINANT real form, not an edge case.
- **How to apply:** (1) Before scoping a feature that reads `legends.yaml`/`cultures.yaml`/`history.yaml`/etc., grep the genre loader for ALL the branches that load that file and enumerate the authoring forms; cite them in the context's Assumptions. (2) Require a fixture per form. (3) In spec-check, don't trust a green suite — scan live content for the authoring shapes the fixtures DON'T cover, then check the code handles them. (4) Reuse the loader's existing dir-vs-flat logic; never re-implement one branch. My 65-12 context wrongly claimed `_load_legends_flexible` "handles both" the directory and flat forms — it does NOT; the directory branch is separate, inline in `_load_single_world`. Verify loader claims against the actual call site, not memory.
