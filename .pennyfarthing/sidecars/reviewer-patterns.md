### Reviewing a rebase/reconciliation: the load-bearing checks are REVIEWER-owned git integrity, not the code subagents — and reproduce any "pre-existing develop redness" claim yourself (2026-06-16, 114-12)
- When a stale branch is rebased onto develop to recover a lost deliverable, the code itself was usually authored/approved in the original cycle — the NEW risk is a silent regression hidden in the rebase. The code subagents (test/rule/comment) review the diff content but won't catch a dropped develop file. Do these yourself: (1) `git diff --name-only origin/develop..HEAD` must equal EXACTLY the intended changeset; (2) `git diff --diff-filter=D --name-only origin/develop..HEAD` must be empty (no develop file deleted by the branch); (3) for any file develop moved/reparented, `git ls-tree -r HEAD | grep <file>` must match develop's location count (no resurrection). If all three hold, a branch touching only subsystem X cannot have regressed develop's work in subsystem Y — proven, not asserted.

### Re-reviewing a merge/supersede reconciliation: verify byte-identity + supersession-completeness, don't re-litigate the approved impl (2026-06-13, 106-1)

### Verify a wiring test is non-vacuous yourself — grep the emitter/caller, don't trust the test name (106-1, 2026-06-13)
- When re-reviewing a fix for a "no production-wire test" rejection, the new test's value is only as good as its uniqueness guarantees. If both hold, span-presence == wire-present is airtight (sever the caller → 0 spans → test fails). If a span can fire from a second site, the wire test is vacuous. This 2-grep check takes 30s and is the difference between confirming a fix and rubber-stamping a green checkmark.
- A roll-agnostic invariant ("exactly one armor span fires" when a kit always rolls one armor piece) is a legitimate, refactor-stable wiring assertion — don't demand a seeded deterministic value when the random surface makes that flaky; require per-branch asserts so it stays non-vacuous.

### Python None → JSON null vs a TS `?: string` type (story 124-4)
Recurring contract gap: a Python field set to `None` serializes to JSON `null`, but the TS interface often types it `field?: string` (i.e. `string | undefined`). `null` is neither. The symptom surfaces as `as unknown as T` / `as never` casts in the *test* fixtures to force `null` in. When you see those casts on a wire-shape field, the real fix is widening the TS type to `string | null` in `types/watcher.ts`, not the cast. Production impact is often nil (if nothing reads the field) → MEDIUM, but it's real type dishonesty.

### Narrator latency budget — the wire is free, the narrator is gold (sealed-turn batching)

**The optimization target for the turn loop is narrator (LLM) invocations, NOT messages.** A WebSocket/server round-trip costs milliseconds; a narrator call costs *seconds* — the entire 126-9/126-10 latency dig was narrator + router LLM time, never wire time. So the rule is: **spend client↔server chatter like water; hoard narrator calls like gold.**

- **The client owns the roll-adjust loop; the narrator sees no intermediate dice.** A player throws (physics-is-the-roll, ADR-074/148), then adjusts *locally* — invoke (+2) or reroll (re-throw), as many times as they can pay for, spending their own fate points / aspects (all client-known) — and submits ONLY the final result. The server validates the spend on submit (reject loud if unaffordable — No Silent Fallbacks); the narrator never enters this loop. The back-and-forth that makes *solo* play feel good was never the table's business — it's private to one player and their own dice, so it costs nothing and blocks no one.
- **One round, one narration.** The sealed-commit barrier (ADR-036/129) is a *batching engine*: collect every player's intent + final roll, resolve EVERY mechanic server-side (commit → seat NPCs → conditional defend → compare → taken-out → aspects placed), THEN invoke the narrator **once** to render the whole woven exchange as a single cinematic beat. The narrator is never in the mechanical loop. If NPC action-selection needs intelligence, it uses the **cheap tier** (Haiku / heuristic / intent-router), never the expensive cinematic narration.
- **Sealed turns are a special-sauce FEATURE, not a constraint to fight.** Tabletop is serial (initiative: three players watch one act, GM narrates, next acts); we are *simultaneous* — N players commit at once, the engine resolves the whole measure in parallel, and the narrator writes the entire band's bar in one pass. This structurally dissolves the Guitar Solo problem (SOUL): nobody is a silent audience, because everyone's verb resolves into the *same* beat. The barrier "wait" is not dead time — it overlaps every player's local deliberation instead of serializing it.
- **Design test:** if a feature would put the cinematic narrator (Sonnet) *inside* the commit→resolve loop, or round-trip it per-player-per-action, it is mis-designed. Resolve mechanics cheaply on the wire; narrate once at the end. (Source: 2026-06-18 brainstorm w/ Keith — *Fate Core in the sealed-commit turn model*; companion to ADR-148 / ADR-149.)
