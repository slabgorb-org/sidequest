## Reviewer Patterns

### Reviewing a rebase/reconciliation: the load-bearing checks are REVIEWER-owned git integrity, not the code subagents ‚Äî and reproduce any "pre-existing develop redness" claim yourself (2026-06-16, 114-12)
- When a stale branch is rebased onto develop to recover a lost deliverable, the code itself was usually authored/approved in the original cycle ‚Äî the NEW risk is a silent regression hidden in the rebase. The code subagents (test/rule/comment) review the diff content but won't catch a dropped develop file. Do these yourself: (1) `git diff --name-only origin/develop..HEAD` must equal EXACTLY the intended changeset; (2) `git diff --diff-filter=D --name-only origin/develop..HEAD` must be empty (no develop file deleted by the branch); (3) for any file develop moved/reparented, `git ls-tree -r HEAD | grep <file>` must match develop's location count (no resurrection). If all three hold, a branch touching only subsystem X cannot have regressed develop's work in subsystem Y ‚Äî proven, not asserted.
- **Stale LOCAL develop bites diff scope:** reviewer-preflight reported "11 files" because it diffed against a stale local `develop` ref (behind `origin/develop`). Always anchor on `origin/develop`, and reconcile any subagent file-count that disagrees before trusting it.
- **Never accept "those failures are pre-existing on develop" on faith** ‚Äî `git switch --detach origin/develop` and run a representative failing test yourself. Here, `test_45_20_trope_resolution_wire.py` failed 10/10 on untouched origin/develop, confirming the 272 full-suite failures were develop's debt (a real-SDK-in-tests hermeticity hole that can bill live API calls), not the rebase's. Scope the green-check to the actual diff + its own tests; flag the develop redness as its own blocking fire.

### "RED until X" docstrings rot into stale GREEN-contradicting markers once X ships ‚Äî flag them [DOC][LOW], and scope-gate untouched vs. touched (2026-06-15, 120-5)
- TEA writes tests with docstrings like "RED until Dev promotes mount_slots to first-class" / "RED today ‚Äî weapons ship damage: None". Once the implementing story lands, those markers are stale: the test is GREEN but the docstring still says it should be failing. comment-analyzer catches them at HIGH confidence and they're real ‚Äî but they're [DOC][LOW] (zero behavioral impact), so they do NOT block per the severity rubric.
- **Scope discipline:** separate markers in code the diff TOUCHED from markers in untouched code. In 120-5 the dev edited one docstring and left its stale "RED until mount_slots parses" line directly above a new GREEN paragraph (self-contradicting ‚Üí confirm in-scope, log it); three other "RED" markers sat in untouched functions (pre-existing debt ‚Üí defer to a follow-up, don't reject the story over them). Approve, but log all of them as non-blocking delivery findings so nothing slips through undocumented.
- **Don't reject green test-hardening over a comment.** Adversarial ‚âÝ disproportionate. Catch it, document it, approve. A REJECT bounces to TEA/Dev for a one-line docstring edit ‚Äî not worth the round-trip when severity is Low.

### Re-reviewing a merge/supersede reconciliation: verify byte-identity + supersession-completeness, don't re-litigate the approved impl (2026-06-13, 106-1)
- When a previously-APPROVED branch is re-sent to review only because it merged develop and superseded a competing duplicate implementation, the review question is NOT "is this implementation good" (already settled) ‚Äî it's three mechanical checks: **(1) did the merge preserve the approved code unchanged?** `git diff <last-approved-sha> HEAD -- <impl files>` must be EMPTY; an empty diff is hard evidence the prior approval still holds. **(2) was the competitor superseded with zero orphans?** Confirm the loser's net-new additions are gone (`chargen.py ‚àí47`, deleted test `‚àí563` showed in the stat) AND grep the tree for the loser's unique symbol/span-name STRINGS (not just Python constant names) ‚Äî zero hits. **(3) is the suite green against the MERGED tree?** The full run exercises the develop-integration far better than reading files.
- Use `git diff origin/develop...HEAD`, NOT `git diff develop...HEAD` ‚Äî the local `develop` branch is routinely stale and the three-dot merge-base then includes everything merged since, inflating the review surface with other people's already-merged work (here it pulled in #835's ~24-file region/movement diff). Always diff against `origin/<default>`.
- Deleting an already-merged competitor's tests is NOT automatically coverage loss: if those tests assert the loser's vocabulary (span names/attrs that no longer exist), keeping them was never an option. Check whether the load-bearing test (the one that was the original reject basis) survives in the winner's suite ‚Äî if yes, the structural defense is intact and the deleted tests' gaps are Medium fix-forwards, not blockers.

### Canonical-snapshot lost-update pattern (epic 73)
- **The identity chain that makes a WRITE-tool persistence fix correct:** `room._snapshot` (what `room.save()` writes, `session_room.py:297`) IS `sd.snapshot` because `connect.py:650/690` reassigns `snapshot = room.snapshot` right after `bind_world`; that same reference flows `sd.snapshot ‚Üí TurnContext.snapshot` (`session_helpers.py:762/1209`) `‚Üí ToolContext.snapshot` (`orchestrator.py:3920`). A WRITE tool that mutates `ctx.snapshot` in place rides the single end-of-turn `room.save()`. Verify every link is a *reference assignment*, not a copy/`model_copy`/re-deserialize ‚Äî that's the whole ballgame.
- **`ctx.repository.load()` returns a FRESH object, never the canonical.** `PgSaveRepository.load() ‚Üí load_snapshot()` (`pg/snapshot.py:134-163`) does `SELECT snapshot_json` + `json.loads()` every call. So the old `load()‚Üímutate‚Üísave()` tool idiom writes a copy the end-of-turn `room.save()` clobbers = silent lost-update. The fix is mutate-canonical + delete-the-in-tool-save.
- **Known latent gap (filed under epic 73):** ~16 other WRITE tools (`apply_damage`, `apply_status`, `advance_encounter_beat`, `apply_spell_effect`, `apply_world_patch`, `commit_*`, `long_rest`, `tick_tropes`, `update_*`, `veterans_luck`, ‚Ä¶) still use the `load()‚Üímutate‚Üísave()` fresh-copy idiom. They survive only if `narration_apply` re-applies their effect to canonical ‚Äî unverified. When reviewing any of these, check whether the effect also lands on the canonical snapshot, or it's the same lost-update class.
- **Opposite-bug check when an in-tool save is *removed*:** confirm an end-of-turn save ALWAYS fires on every production path that can invoke the tool (`websocket_session_handler.py:1171` persistence phase + `:454` disconnect). Removing the save is only safe if the tool is the anomaly and the rest of the turn already relies on the single canonical save.
- **`canonical=True`-style OTEL signals are presence-only, not booleans.** A hardcoded-`True` attribute on the success path that's simply *absent* on the fail-loud path is the right lie-detector shape ‚Äî but flag docstrings that frame it as distinguishing two values (it can never be `False`).

### Verify a wiring test is non-vacuous yourself ‚Äî grep the emitter/caller, don't trust the test name (106-1, 2026-06-13)
- When re-reviewing a fix for a "no production-wire test" rejection, the new test's value is only as good as its uniqueness guarantees. For an OTEL-span wire test ("assert exactly one of {span_A, span_B} fired through the real handler"), independently confirm: (1) the asserted spans are emitted from EXACTLY ONE function (`grep` the span-helper call sites, excluding the telemetry def module), and (2) that function has EXACTLY ONE production caller (`grep` the function name, exclude tests). If both hold, span-presence == wire-present is airtight (sever the caller ‚Üí 0 spans ‚Üí test fails). If a span can fire from a second site, the wire test is vacuous. This 2-grep check takes 30s and is the difference between confirming a fix and rubber-stamping a green checkmark.
- A roll-agnostic invariant ("exactly one armor span fires" when a kit always rolls one armor piece) is a legitimate, refactor-stable wiring assertion ‚Äî don't demand a seeded deterministic value when the random surface makes that flaky; require per-branch asserts so it stays non-vacuous.

### Python None ‚Üí JSON null vs a TS `?: string` type (story 124-4)
Recurring contract gap: a Python field set to `None` serializes to JSON `null`, but the TS interface often types it `field?: string` (i.e. `string | undefined`). `null` is neither. The symptom surfaces as `as unknown as T` / `as never` casts in the *test* fixtures to force `null` in. When you see those casts on a wire-shape field, the real fix is widening the TS type to `string | null` in `types/watcher.ts`, not the cast. Production impact is often nil (if nothing reads the field) ‚Üí MEDIUM, but it's real type dishonesty.
