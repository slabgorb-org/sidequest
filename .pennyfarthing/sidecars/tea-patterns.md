### Core wiring patterns
- **Wiring means non-test consumers.** Before declaring GREEN, grep for the function/struct in production code paths. Library functions with no consumers are stubs, not features.
- **Wiring means the full pipeline.** "Wire X into Y" = source â†’ intermediate layers â†’ rendered output, not "component accepts props." ACs and tests must cover the full data flow.
- **Wiring means visible in the GM panel.** Every subsystem emit site gets an OTEL watcher event with content the lie detector needs to refute a Claude hallucination (e.g. `room_count`, `hp_delta`, `item_added`). Internal data flow without OTEL is not wired.
- **Combat wiring trace (reference):** IntentRouter â†’ Orchestrator â†’ lib.rs â†’ Protocol â†’ UI CombatOverlay. This is the template for verifying any intent-to-UI pipeline.
- **Debug, don't rebuild.** The code is ~98% done â€” find the small break. Trace the existing pipeline front-to-back before writing anything new. Dev agents repeatedly rebuild pipelines that already work.
- **Source-level wiring guard tests.** When a runtime integration test is impractical (dispatch_connect has 55 params, dockview needs real DOM sizing, bool CLI defaults), read the source file from the test and assert the wire exists as a substring/pattern within N lines of the emit site. Pattern reference: `map_telemetry_wiring_tests.rs` in sidequest-server, `gameboard-wiring.test.tsx` in sidequest-ui. These catch "someone deleted the wire in a future refactor" at `cargo test` / `vitest run` time, not at playtest time.

### Architectural patterns
- **Rust vs Python split.** If it doesn't operate an LLM, it goes in Rust (game engine, state, protocol, markov chains, name gen). If it runs model inference (Flux, Kokoro, ACE-Step â€” *not* Claude), it's in the Python daemon. Claude CLI calls go through Rust subprocess.
- **Sidecar tool pattern** (Epic 20): Narrator calls tool â†’ parser validates â†’ `assemble_turn` consumes. Established by `item_acquire`, `merchant_transact`, `lore_mark`. All future narrator-exposed tools follow this pattern.
- **Script tool prompt pattern** (ADR-056): Expose Rust generator binaries to the narrator via skill-style checklists (carrot), not threatening gate language (stick). Clear command table, "when to call" guidance, post-call checklist. Never "You MUST" or "Do NOT" â€” LLMs bash against hard gates.
- **No server-side re-invocation of the LLM to judge the LLM.** Post-hoc validation lives in OTEL spans ("was the tool used? did output match state?"), not a second LLM round-trip. The user calls this the "God lifting rocks" problem.
- **React `key` as layout reset tool.** Changing `key` on a stateful component forces React to unmount and remount it fresh â€” cleanest way to reset dockview/three.js/etc. without imperative reset code. Used in `GameBoard key={currentGenre}` to fix the Dockview drift bug.

### Git / branch patterns
- **Gitflow on every subrepo.** Subrepos (sidequest-server, sidequest-ui, sidequest-content, sidequest-daemon) target `develop`. Only the orchestrator targets `main`. Always check `repos.yaml` before any git op.
- **Branch before editing.** `git checkout -b feat/description` BEFORE touching files, not after. Avoids the "hook-blocked commit â†’ panic â†’ stash scramble" failure mode.
- **Dirty work comparison pattern.** When local uncommitted changes conflict with a pull: commit dirty work to a temp branch, pull clean on the default branch, diff each file local-vs-remote side by side, categorize (identical / remote better / local has extra value / no remote change), then cherry-pick deliberately. Never stash.
- **Additive git ops need no permission** (commit, push non-force, pull --ff-only, checkout existing, checkout -b, fetch, add). **Destructive git ops always ask first** (reset, clean, force-push, branch -D, rebase -i, stash â€” banned outright).

### Narrator latency budget â€” the wire is free, the narrator is gold (sealed-turn batching)

**The optimization target for the turn loop is narrator (LLM) invocations, NOT messages.** A WebSocket/server round-trip costs milliseconds; a narrator call costs *seconds* â€” the entire 126-9/126-10 latency dig was narrator + router LLM time, never wire time. So the rule is: **spend clientâ†”server chatter like water; hoard narrator calls like gold.**

- **The client owns the roll-adjust loop; the narrator sees no intermediate dice.** A player throws (physics-is-the-roll, ADR-074/148), then adjusts *locally* â€” invoke (+2) or reroll (re-throw), as many times as they can pay for, spending their own fate points / aspects (all client-known) â€” and submits ONLY the final result. The server validates the spend on submit (reject loud if unaffordable â€” No Silent Fallbacks); the narrator never enters this loop. The back-and-forth that makes *solo* play feel good was never the table's business â€” it's private to one player and their own dice, so it costs nothing and blocks no one.
- **One round, one narration.** The sealed-commit barrier (ADR-036/129) is a *batching engine*: collect every player's intent + final roll, resolve EVERY mechanic server-side (commit â†’ seat NPCs â†’ conditional defend â†’ compare â†’ taken-out â†’ aspects placed), THEN invoke the narrator **once** to render the whole woven exchange as a single cinematic beat. The narrator is never in the mechanical loop. If NPC action-selection needs intelligence, it uses the **cheap tier** (Haiku / heuristic / intent-router), never the expensive cinematic narration.
- **Sealed turns are a special-sauce FEATURE, not a constraint to fight.** Tabletop is serial (initiative: three players watch one act, GM narrates, next acts); we are *simultaneous* â€” N players commit at once, the engine resolves the whole measure in parallel, and the narrator writes the entire band's bar in one pass. This structurally dissolves the Guitar Solo problem (SOUL): nobody is a silent audience, because everyone's verb resolves into the *same* beat. The barrier "wait" is not dead time â€” it overlaps every player's local deliberation instead of serializing it.
- **Design test:** if a feature would put the cinematic narrator (Sonnet) *inside* the commitâ†’resolve loop, or round-trip it per-player-per-action, it is mis-designed. Resolve mechanics cheaply on the wire; narrate once at the end. (Source: 2026-06-18 brainstorm w/ Keith â€” *Fate Core in the sealed-commit turn model*; companion to ADR-148 / ADR-149.)
