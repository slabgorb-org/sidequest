### Companion brain telemetry (161-2): centralize the emit in brain.decide() via an optional defn, offload the sync bridge, gate the server passthrough (2026-07-01, 161-2)
- **Emit lives in `companion/brain.py::decide()`, NOT `run.py` — via an OPTIONAL `defn` kwarg.** `decide()` is the single `asyncio.wait_for` chokepoint; it already has the `DecideResult` + does the timing. Adding `*, defn: CompanionDef | None = None` there (a) covers ALL four `run.py` decide() call sites (TURN_STATUS, PROMPT_KINDS, two chargen) uniformly, and (b) keeps the 4 pre-existing `test_brain.py` tests green — they call `decide(...)` WITHOUT defn → no emit. `run.py` threads `defn=defn` into every call. Amos's RED tests patched `companion.telemetry._post` (the network boundary), so they pass regardless of whether emit lives in run.py or brain.py — the freedom was deliberate. One emit site beats three.
- **The sync urllib POST is offloaded via `await asyncio.to_thread(emit_watcher_event, ...)` AND wrapped in a guard try/except that logs+swallows.** The bridge mirrors the daemon's SYNC `watcher_bridge.py` (stdlib urllib, 2s timeout) but the companion loop is ASYNC — a raw inline call would block the event loop for up to 2s and stall the table (lang-review #9). to_thread fixes that; the outer guard means even a non-network emit bug can't break the companion's move (AC #5). The bridge itself already swallows `(URLError, ConnectionError, OSError, TimeoutError)` → `log.warning`; the guard is belt-and-suspenders for everything else.
- **`asyncio.TimeoutError` must be caught BEFORE the broad `except Exception`** to set `timed_out=True` distinctly from other errors (both degrade to YIELD, but the telemetry distinguishes them). `except asyncio.TimeoutError:` then `except Exception:`. A non-CompanionIntent `.value` (valid call, unusable decision) also sets `degraded=True` — silence would hide a model failure from the lie-detector.
- **Server passthrough GATES the optional args so the daemon path is byte-identical.** The `/internal/watcher/emit` handler builds `extra={}` and only adds `session_slug`/`severity` when the payload supplies them, then `publish_event(..., **extra)`. This is load-bearing: the existing `test_watcher_bridge_endpoint.py` asserts `publish_event` is called with EXACTLY `event_type/fields/component` — always passing `session_slug=None` would break it, and passing `severity=None` would clobber publish_event's `"info"` default (the daemon guard test catches that). `publish_event` itself: added `session_slug: str | None = None` (keyword-only) and `slug = session_slug or current_session_slug()` (override beats ContextVar; None → unchanged).
- **DECISION FLAGGED: anthropic `cost_usd` stays 0.0.** The Anthropic SDK response carries no per-call price (only tokens); computing cost would need a hardcoded per-model price table (fragile, drift-prone) that no test demanded. `claude_p` is the cost-metered backend (`total_cost_usd` in its envelope); ollama is a real 0.0 (free/local). Since the companion's DEFAULT_MODEL is `anthropic/...`, most decisions report cost 0.0 — flagged as a non-blocking Delivery Finding + a deviation so Reviewer/Keith can rule on whether a price table is worth it. The 161-3 UI already labels claude_p cost "plan-equivalent (notional)" vs anthropic "metered", so the surfaces distinguish them.
- **anthropic `input_tokens` KEEPS summing cache** (`input + cache_read + cache_creation`) — the existing `test_anthropic_forces_tool_caches_prefix_and_meters_true_input` pins `10+200+30`. `cache_tokens` is surfaced SEPARATELY (`cache_read + cache_creation`) for the ledger without touching the token-ceiling meter. Don't "fix" input_tokens to stop summing — it'll break that test and the ceiling semantics.

### Measure, don't assert — no runtime root cause from indirect evidence (2026-05-22, 60-3)
- When debugging runtime behavior (cache TTLs, model routing, what the server actually sends), never assert a conclusion from indirect evidence. Code defaults, token-count archaeology, and a couple of stale log lines are hypotheses to test, not findings to announce. In the 60-3 cache diagnosis a root cause was declared 4× and each overturned (e.g. "narrator is on Sonnet" read from `_DEFAULT` map, never logged `response.model`).
- Measure the actual artifact: log `response.model` from the live call, `cache_creation.ephemeral_5m/1h` from the real request, the real dicts sent to the SDK.
- Isolate one variable with a minimal reproducer (the `/tmp/ttl_probe.py` approach). When probe diverges from prod, the cause is in the unreplicated difference — enumerate and close those.
- State confidence honestly ("Measured: X. Unconfirmed: Y"). No "damning"/theatrics. Stale logs mix sessions/builds — mark an offset, run fresh, read only new lines.

### Don't bury bombs with null-checks on a shared init path (2026-05-06, playtest)
- When a shared init path (e.g. chargen-completion) raises AttributeError because a special-case world/data-shape leaves expected fields None, do NOT add per-site null-guards to silence the crash — that buries the bomb: downstream subsystems silently no-op and the narrator runs against undercooked state (the "Claude wings it" anti-pattern).
- Incident: hub-world `caverns_three_sins` had no top-level cartography → `init_region_location` AttributeError'd. Null-guards cleared the crash but skipped magic init, chassis registry, scenario binding, opening narration (the earlier blank opening was the consequence). Fix was reverting the hub-world special case to a normal leaf world.
- Two acceptable responses: (1) restore the data so the special case is no longer special, or (2) own a complete end-to-end init pipeline for the special case. Adding null-checks one crash at a time is bomb-burial — never the right move.

### Banned: git stash, and "verify failure is pre-existing" forensic checks (2026-05-10, from Keith)
- Never use `git stash` in any form (`-u`, `pop`, `--keep-index`, etc.) — stashes get lost and are invisible in `git branch`. Commit WIP to a feature branch instead (`git checkout -b wip/<topic>`, commit, push, switch back).
- Never check out a prior commit / use worktree / `git show` / `git checkout` to prove a failure is pre-existing. The user doesn't want forensic verification of pre-existing failures. Just report "regression: N pre-existing failures, not related to this change" and move on.
- Both prohibitions apply to every dispatched subagent — include them explicitly in subagent prompts (project memory doesn't propagate). Watch the slip pattern of chaining `git stash --keep-index` inside a `;`/`&&` bash chain to disguise it.

### `claude -p` cannot do reactive tool calls during generation — a transport constraint under ADR-001 (2026-05-02, from Keith)
- `claude -p` is a one-shot subprocess returning a single text response. There is NO execution model under ADR-001 that lets the narrator call `set_mood` at line 4 and reference its return value at line 7. The `--allowedTools` flag exists but does not give `claude -p` the streaming tool-then-resume-generation loop the Anthropic SDK supports.
- Any narrator-architecture ADR/design that has the narrator calling `set_X`/`lore_mark`/`item_acquire` tools DURING generation is dead on arrival — flag immediately. ADR-057 specified exactly this (narrator-calls-tools-mid-generation + post-narration `assemble_turn`) and was infeasible from inception, never implemented.
- The only way the narrator emits structured data is by writing it directly into its single text response — a fenced `game_patch` JSON block (`sidequest-server/sidequest/agents/narrator.py:80`, the `NARRATOR_OUTPUT_ONLY` constant). Server-side tool calls (orchestrator invoking namegen/encountergen/loadoutgen BETWEEN turns per ADR-059) are fine — they run outside the narrator's generation context. If a future design genuinely needs reactive tools, ADR-001 itself (CLI → SDK) must be revisited.

### Fate-SRD genres — preserve vs. drop native behavior: the governing axis (Keith, 2026-06-17)
- **The native vision is being retired** (dials/beats/edge-tags were fun but un-balanceable, and homebrew authors can't write against native — they need an SRD to write against). When converting a genre to a bound SRD (Fate/WN), and you hit a "should we honor / port the native behavior?" call, decide by **which layer the behavior belongs to:**
  - **Multiplayer / remote-substrate mechanics → PRESERVE, and make the SRD engine bend to accommodate them.** These exist *because* SideQuest is remote with no human at a table and no human GM, so the system must track things a tabletop GM just knows. Examples: the **sealed-turn / all-submit-at-once barrier** (wait for everyone), **who is present / who acted this turn** (ADR-72-12 presence stamping + `npc.edge_published`, so the 72-6 last-seen prune doesn't drop an actively-engaged NPC), the **universal `encounter.resolved` signal** (panel teardown, input unlock, render trigger, the forensic-timeline / GM-panel lie-detector row — `watcher_hub` maps `op:"resolved"`→`ENCOUNTER_RESOLVED`), and **ADR-116 "a confrontation requires an Other"** (seating + fail-loud). Fate has to allow these even though tabletop Fate doesn't need them.
  - **Genre *resolution* mechanics → the SRD owns them; do NOT port native.** How an opposed roll actually resolves — dials, threshold-7 calibration (ADR-093), beat `stat_check`s, native confrontation math — is replaced by the SRD (Fate Contest/Conflict; WN combat). In a Fate genre, opposed rolls go through Fate. Deleting the dial-advance / opposed_check-mode / threshold / opponent-stat-ceiling assertions for a converted pack is correct.
- **Litmus:** "Is this behavior here because we're remote-with-no-GM, or because it's how the old native rules computed a result?" Substrate → keep (bend the SRD). Resolution → drop (SRD owns it).
- **Reconciles with bind-don't-balance (SOUL.md / ADR-143):** preserving presence-stamp + resolved-span on the Fate Contest path is NOT re-introducing native balancing — these are SideQuest-side observability/world-state/coordination concerns, not Fate math. We're wiring the platform substrate through the new engine, not tuning Fate against native.
- **Applied in Task 12b** (fate-contest-binding regression fix): GROUP A native-resolution assertions DELETED (rescope only the still-native road_warrior ones); GROUP B substrate invariants (seating/fail-loud/presence/resolved-span) PRESERVED via the contest path — incl. adding per-exchange presence-stamp + `encounter_resolved_span` to `run_fate_contest_exchange` (`fate_contest.py`), gated to the Fate path by construction.

### Per-turn narrator cost: static prose is squeezed — RAG is the next lever (2026-06-06, Keith directive)
- **Decision:** **Do not re-attempt trimming the stable narrator prose** (`sidequest-server/sidequest/agents/narrator_prompts/*.md` — the `STABLE_SECTION_NAMES` set in `prompt_framework/bucket.py`). That juice is squeezed. The next per-turn cost lever is **proper RAG utilization to shrink the volatile `game_state` snapshot**, not prose edits.
- **Why (the forensics):** A live `2026-06-06-the_circuit` drive showed the per-turn structure is already correctly tiered: a ~14.3k-tok **stable prefix** (system + tools) written once @1h and read at $0.30/M, and a ~10–12k-tok **volatile tail** written @5m (~$0.04/turn). The volatile tail is dominated by the `game_state` snapshot (~14.5KB / ~3.6k tok), which is **truncated** to fit — 14+ NPCs dropped per turn — while the ADR-118 universal-retrieval layer is **already live** and surfaces the relevant NPCs/factions/lore by similarity for ~300–700 tok. We pay twice: fat snapshot *plus* retrieval. The static `.md` prose (~5.5k tok total) is in the **read** tier — a 20% cut ≈ $0.0003/turn (~$0.03 / 85-turn session): wrong tier, negligible dollars.
- **Why trimming the prose is actively unsafe:** what reads as "redundancy" is mostly **test-pinned regression fingerprints**. Compressing the four "Patients on a sickbed count…" sentences in `output_only.md` broke `tests/agents/test_57_4_recency_guardrails_migration.py` and `test_61_12_output_format_compaction.py` — and **ADR-111 §Alternatives B explicitly rejected** exactly that compression ("the bug-report specificity is the regression detector, not stylistic flavor"). Add primacy/recency bookends + concrete examples (e.g. the #718 "behind him" POV case) that drive LLM compliance. This is the surface that has to "fool a career GM"; the trade is narrator quality for fractions of a cent.
- **How to apply:** For per-turn cost work, go after the `game_state` snapshot via the **RAG-rebalance** — lean on ADR-118 retrieval so the snapshot carries only structural essentials (PC, current location, active confrontation/dials) instead of shipping-then-truncating the full roster. This is ADR-110's deferred phase C (diff-with-anchor) territory and an architecture call (discuss with Keith — it's mechanics/crunch-adjacent). Full analysis: `docs/analysis/2026-06-06-narrator-per-turn-cache-write-anatomy.md` and `…-narrator-prompt-verbosity-trim-report.md`. Don't redo Story 60-3/60-4/61-19/61-20 (cache tiering) or 57-3/57-4 (prose promotion + fingerprints) — those are done.

### A green-exit that surfaces large pre-existing full-suite debt: FILE it, keep the branch tight (2026-07-01, story 160-4, Keith directive)
- **Decision:** When "make the suite green" surfaces failures that are NOT your story's tests, sort them by one cut: **"is this failure blocking verification of MY story?"** If yes (a shared fixture/helper the story's own tests ride on), fix it and **log it as a deviation** — it's in-scope hygiene, not scope creep. If no (deep pre-existing engine/architecture/test-infra debt), **file it as its own scoped story with the diagnosis and land your story tightly scoped.** Do not drag unrelated debt onto a feature branch; it bloats the review surface and couples unrelated risk.
- **Why:** On 160-4 (companion-solo-seat) the full suite had ~23 pre-existing reds. Two were shared-fixture drift the 160-4 wiring test itself needed — `session_fixture.effective_bestiary` had to return the 158-33 `(Bestiary|None, str)` 2-tuple, and `husk_reaped_this_turn` had to join `_EXCLUDED_FROM_DUMP` — so those were fixed on-branch and logged as deviations. - **Hard guardrail (Keith, restated as doctrine):** fix real ROOT CAUSES, but **never gut assertions or weaken a deliberate fail-loud guard to force green** (e.g. do not soften the `EncounterSeedError` at `pregen.py:587`). A green suite bought by deleting the detector is worse than a red one. Forcing green that way IS the naive regression.
- **How to apply:** at green-exit, run the full suite once, bucket the reds by (a) blocks-my-verification vs (b) pre-existing-unrelated; fix (a) + log as deviations, file (b) as stories under the active playtest-follow-up epic with concrete evidence + fix direction; record both in the session Delivery Findings so the Reviewer sees the trail. Confirm the file-vs-fix scope with Keith before grinding open-ended engine debug into a feature branch — he steers these forks.
