## Problem

After F2a, the **router** sees Fate state (skills, fate points, aspects) via `_build_fate_summary`, so it can classify freeform actions. But the **narrator** is blind: it gets no Fate state in its prompt, so it cannot name a character's aspects, cannot tell the player which aspects are *invokable* this turn, and has no channel to *propose a compel*.

## Technical Approach

F2b closes the narrator side of the gap with three additions and zero new engines, per `docs/superpowers/plans/2026-06-14-f2b-fate-narrator-aspects.md`:

1. **Extract the projection to one source of truth** — relocate `_build_fate_summary` from `intent_router_pass.py` to a new `game/ruleset/fate_projection.py` as `build_fate_projection(snapshot: GameSnapshot) -> dict[str, Any]`, so both router and narrator import it cleanly from the `game` layer.

2. **Register a Fate prompt section** — in `build_narrator_prompt` (orchestrator.py:1746), add a Late/State section that renders the projection + an invokable-aspect directive instructing the narrator to *propose* invokes (not auto-spend). Gated on `pack.rules.ruleset == "fate"`.

3. **Wire the `propose_fate_compel` tool** — new `agents/tools/fate_tools.py` with a thin WRITE tool that calls the existing `FateRulesetModule.offer_compel`, firing the existing `fate.compel.offered` span. Tool is ruleset-gated to Fate packs only via the advertisement filter.

## Scope
- In scope: the three additions above (projection move, prompt section, tool wiring).
- Out of scope: compel acceptance (F3 UI); create-advantage rendering (F2c); opponent AI (F2d).
