# Story 153-26 Context

## Title
[DUNGEON-CURATE-TIMEOUT] keep dungeon room curate within the 25s wall-clock cap (precurate/stream) so authored room content and encounters are not dropped to the degraded deterministic manifest

## Problem
The ADR-106 runtime procedural megadungeon (caverns_and_claudes/beneath_sunden) generates
and edge-expands rooms at runtime. Each room runs a **curate** pass that fleshes out authored
room content and encounters. That curate pass can exceed a ~25s wall-clock cap; on timeout the
system falls back to a **degraded deterministic manifest**, which silently **drops authored room
content and authored encounters**.

Observed in the 2026-06-20/21 playtest forensics on beneath_sunden: a hand-authored
`gnaw_swarm` encounter was dropped because curate timed out and the deterministic fallback was
served instead. Net effect: content the world designer placed never reaches the player, and the
drop is silent (no loud signal that the degraded path was taken).

## Technical Approach (investigation pointers — TEA/Dev to refine; not a prescribed design)
Fix direction per the story title + playtest forensics: keep the per-room curate **within** the
25s cap so the authored path survives instead of degrading. Two named levers in the title:
- **precurate** — do curation work ahead of arrival / off the narrator hot path, so it's ready
  by the time the player enters the room.
- **stream** — incremental curation so partial authored content lands within the cap rather than
  an all-or-nothing timeout.

Likely code areas (sidequest-server): the ADR-106 megadungeon room-generation / curation path
under `sidequest/game/` (room graph, region projection, complication ledger), the curate
wall-clock cap (grep the ~25s constant), and the degraded deterministic-manifest fallback branch.

Per the CLAUDE.md OTEL principle: the degraded fallback must emit an OTEL/watcher span when it
fires, so the GM panel can detect curation degradation instead of silently shipping uncurated
rooms (it currently degrades silently — that's part of the bug).

## Scope
- **In scope:** keeping dungeon room curate within the 25s wall-clock cap (precurate and/or
  stream) so authored room content + encounters survive; a loud (OTEL) signal when the degraded
  deterministic path is taken.
- **Out of scope:** changing the ADR-106 generation/edge-expansion algorithm itself; room-graph
  seam-crossing (that was 105-2 / server #1016, already merged); Map-UI room-graph rendering.

## Acceptance Criteria (candidates — TEA formalizes as failing tests in RED)
1. On a representative beneath_sunden room expansion, curate completes within the 25s wall-clock
   cap and the room is served from the curated (authored) path — not the degraded deterministic
   manifest.
2. Authored room content and authored encounters (e.g. `gnaw_swarm`) are present in the served
   room/manifest — not dropped — when curation runs under the cap.
3. The degraded deterministic-manifest fallback triggers only on a genuine curate timeout/failure,
   and emits an OTEL/watcher span (loud, not silent) when it does.
4. **Wiring test:** the curate-within-cap path (precurate/stream) is reachable from the production
   dungeon room-entry code path — imported, called, and exercised end-to-end — not just in isolation.
