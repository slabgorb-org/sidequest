## Title
QUESTS not re-emitted on connect/resume — mirror the FATE_STATE bootstrap re-emit so the Quests tab survives reload/reconnect

## Problem

This is the **fourth instance** of a recurring pattern in the connect/resume bootstrap: the `_State.Playing` resume bootstrap in `sidequest-server/sidequest/server/connect.py` fails to re-emit something that a per-turn emitter emits. The Quests tab disappears on reload/reconnect because the `QUESTS_STATE` payload (emitted per-turn by the quest subsystem) is not re-emitted when the session resumes after a disconnect.

**Recurring pattern history:**
1. FATE_STATE (153-7) — fixed by adding `_maybe_emit_fate_state()` to the bootstrap
2. LOCATION_DESCRIPTION (153-7) — fixed by adding `_maybe_emit_location_description()` to the bootstrap
3. FATE_DEFEND_REQUEST (153-7) — fixed by adding `_maybe_emit_fate_defend_request()` to the bootstrap
4. QUESTS_STATE (158-15) — to be fixed following the same pattern

## Technical Approach

**The fix pattern (mirror ADR-036 bootstrap recovery):**

1. **TEA (red phase):** Write a failing test proving QUESTS_STATE is NOT re-emitted on reconnect/reload today.
   - Test should verify the Quests tab is empty after a disconnect→reconnect cycle.
   - Include a wiring test that proves the new re-emit fires through the production connect/resume path (not just in isolation).

2. **Dev (green phase):** Add a `_maybe_emit_quests_state()` helper in the resume bootstrap, placed next to the existing FATE_STATE re-emits (in `sidequest-server/sidequest/server/connect.py`, `_State.Playing` resume method).
   - Mirror the FATE_STATE bootstrap re-emit implementation exactly.
   - Call the helper from the resume bootstrap to re-emit QUESTS_STATE upon reconnection.
   - Verify the wiring test passes (production path).

**Reference implementation:** Examine `_maybe_emit_fate_state()` in `sidequest-server/sidequest/server/connect.py` (story 153-7 baseline) — the QUESTS variant follows the same structure.
