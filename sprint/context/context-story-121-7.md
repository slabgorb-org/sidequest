## Problem
The chargen flow that produces explicit Fate choices for F4a's seeding to apply — skill allocation per the chosen method with legality validation (pyramid/column shape), mandatory aspects (high concept + trouble) + free aspects, stunt selection with refresh accounting, ruleset=='fate' chargen mode that does NOT require d20 stats/classes.

## Technical Approach
Build the interactive Fate chargen validator + engine following the design in `docs/superpowers/specs/2026-06-16-fate-interactive-chargen-design.md`:

- **Skill allocation validator:** pyramid legality (column-count check per pack config)
- **Aspect editor:** high-concept + trouble + N free aspects (N pack-configurable)
- **Stunt picker:** with refresh accounting (floor: 1 refresh, default 3 refresh / 3 free stunts)
- **Archetype seeding:** seed-then-edit model (archetype seeds all; player edits + confirms HC+Trouble)
- **Fate chargen mode:** integrated into ADR-015 builder FSM via new scene input_types; no d20 machinery
- **Wire contract:** server → UI payload for Guided/Freeform chargen
- **OTEL wiring:** fate.chargen.* spans (aspect authoring, pyramid allocation, stunt selection)

## Scope
- In scope: 
  - `apply_fate_chargen()` builder path + legality validators for pyramid/aspects/stunts
  - Server wire contract for Guided/Freeform chargen payloads
  - OTEL spans for each chargen step
  - Wiring test: real fate-pack chargen run yields a legal FateSheet through the production builder
- Out of scope: 
  - UI renderers (story 121-8, F4a3)
  - Content authoring (stories 121-3/4/5)

## Acceptance Criteria
_Defined by TEA (The Architect) during RED phase._

- **AC1 — Pyramid legality validator.** A `validate_pyramid_allocation(pyramid: dict[str, int], pack_config: FateConfig) -> bool` function checks skill allocation against pack config (column-count; SRD constraint: one skill at +4, one at +3, two at +2, three at +1, rest at +0). The validator rejects illegal allocations and reports legible errors.
- **AC2 — Aspect editing.** A `validate_aspects(aspects: dict[str, str], mandatory_pair: [high_concept, trouble]) -> bool` function ensures high-concept + trouble are populated and non-empty. Free aspects are optional (pack-configurable count).
- **AC3 — Stunt economy validator.** A `validate_stunt_selection(stunts: list[str], free_count: int, refresh: int) -> (valid: bool, new_refresh: int)` checks that stunt cost does not exceed the refresh budget (each extra stunt costs 1 refresh, floor 1).
- **AC4 — Fate chargen wire contract.** A `FateChargenRequest` payload (archetype, authored aspects, pyramid, stunts) and `FateChargenResponse` (validated sheet, errors) exist; the contract is wired into the builder's Guided/Freeform paths.
- **AC5 — OTEL lie-detector.** Each chargen step emits a span (fate.chargen.aspects_authored, fate.chargen.pyramid_allocated, fate.chargen.stunts_selected) with counts + validation result.
- **AC6 — Wiring (mandatory, no source-grep).** A real fate-pack chargen run (e.g., pulp_noir) driven through the production builder + real validator yields a legal FateSheet with populated pyramid, aspects, and stunts. OTEL/fixture-driven, not a read_text() grep.
