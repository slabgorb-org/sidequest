# Story 118-4 Context

## Title
F3d — Invoke control: player-driven invoke from the Fate panel

## Metadata

## Problem

F3a (FATE_STATE projection) and F3b (Fate panel read surface) are complete and live. F1d/F2a already wired FateActionPayload.invoke_aspect at the server layer. F3d must surface the invocation control to the player: an "Invoke (+2 / reroll)" affordance per aspect showing the cost (free invoke if available, else 1 fate point). The control must execute optimistically on the client, decrementing fate points on submit, and reconcile with the server's next FATE_STATE update. This surfaces the F2b invokable-aspect directive as clickable UI instead of prose.

## Technical Approach

### Scope

- **Client:** Build the invoke affordance in the FatePanel, per-aspect.
  - Render a button/badge per aspect in the aspect list (FatePanel.tsx or AspectItem component).
  - Display cost: "Free" if the aspect has free invokes remaining, else "1 Fate".
  - On click, dispatch FATE_ACTION with kind: 'invoke_aspect', aspect_id, and optionally the invocation mode ('+2' or 'reroll').
  - Optimistically decrement fate.current_fate_points on submit (trust the server as the sole authority; reconcile on next FATE_STATE).
  - Render disabled state if no fate points and no free invokes.

- **Server:** Already done (F1d/F2a). FATE_ACTION handler exists; FateActionPayload.invoke_aspect is valid. Ensure FATE_STATE is emitted post-action to carry the updated fate points and aspect free-invoke counts.

- **Acceptance Criteria**
  - The FatePanel renders an invoke affordance (button/badge with tooltip) for each aspect that is eligible for invocation (aspect.kind in {high_concept, trouble, character, situation}).
  - The affordance displays the cost: "Free Invoke" if aspect.free_invoke_count > 0, else "1 Fate Point".
  - On click, send FATE_ACTION with kind='invoke_aspect', aspect_id={id}, and player_id={authenticated_player_id} (do NOT accept inbound player_id spoofing per 118-8; use the session-authenticated identity).
  - The client optimistically decrements game state's current_fate_points in the store (use a temporary decrement, not a commit) and immediately reflects the change in the panel.
  - After invoking, the panel reconciles with the next FATE_STATE message from the server — if the server's fate_points or aspect counts differ from the optimistic decrement, the panel updates to the authoritative server state (No Silent Fallbacks: log a loud console warning if the reconciliation shows a mismatch).
  - Negative test: an invoke affordance must NOT render for aspects that lack free invokes AND the PC has 0 fate points (disabled state, tooltip explains "no fate points remaining").
  - Negative test: the invoke button is grayed out / disabled during the turn submission phase (sealed-commit barrier per ADR-036) so players cannot invoke while waiting.

- **Out of Scope**
  - Interactive chargen (fate sheet authoring) — deferred to F4.
  - Compel accept/refuse — that is F3e.
  - Conflict overlay — that is F3f.
  - The server-side fate accounting (FateActionHandler already exists; F3d is UI only).

## Architecture Notes

- **Dependency:** 118-2 (FATE_STATE projection + FatePanel read surface) — must be live.
- **Related:** 118-3 (4dF dice display) is adjacent UI work for Fate; not a hard dependency for F3d (different mechanic).
- **Live Branch:** origin has feat/118-3-fudge-die-4df-display (118-3 work) — be aware it may have conflicting CSS or panel structure changes.
- **Guard:** Every invoke surface must be gated on ruleset=='fate' (pack.rules.ruleset check in GameStateProvider); never render invoke buttons for WN/native packs.

## Acceptance Criteria

- An invoke affordance renders per aspect in the FatePanel, showing cost ("Free Invoke" or "1 Fate Point").
- Clicking invoke sends FATE_ACTION with kind='invoke_aspect' and aspect_id; player identity is driven from the session-authenticated identity, never inbound msg.player_id.
- Client optimistically decrements fate.current_fate_points on submit and updates the panel display immediately.
- Client reconciles with the next FATE_STATE from the server; loudly logs any mismatch between optimistic state and server state (No Silent Fallbacks).
- Aspects without free invokes and with zero fate points show a disabled, tooltipped invoke button.
- The invoke button is disabled during turn submission phase (sealed-commit barrier).
- Negative test: invoke button never renders for non-Fate rulesets.
