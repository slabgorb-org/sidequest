Server companion bond + perception seam

## Problem
The server must recognize when a WebSocket seat is bonded to a companion and scope its perception based on the companion's role type (pet/peer/hireling). A pet should receive owner-private information (NARRATION_SEGMENT, SECRET_NOTE), while a hireling should see only what's publicly observable. Unknown relationships must fail closed.

## User Story
As a player, I want the server to recognize a seat as a companion bonded to me and scope its perception by type, so that a bonded pet shares my private view while a hireling sees only what's observable.

## Technical Approach

Implement the companion bond and perception seam per Plan B in docs/superpowers/plans/2026-06-25-companion-B-server-bond-seam.md:

1. **Bond Registry** — Store companion → owner bonds in session state with bond metadata (role type: pet/peer/hireling)
2. **Connect-Handler Registration** — Wire the SESSION_EVENT connect handshake to accept companion metadata and resolve the bond
3. **Fan-Out Pet-Widening** — In the broadcast-layer perception firewall (ADR-105), check companion bond status before scoping per-seat perception

Wire this into existing infrastructure:
- **Perception Firewall:** ADR-104 (tool layer) + ADR-105 (broadcast layer) — do NOT reimplement, integrate
- **MP Item Attribution:** ADR-108 — leverage existing per-recipient tagging
- **Disposition/OCEAN Machinery:** ADR-020, ADR-042 — reference for character context if needed
