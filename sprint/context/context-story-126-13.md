## Problem
Follow-up to 126-8 (Fate DEFEND barrier, merged as PR #954). When a player
attempts a defend-path action and the seat/authorization check **rejects** it
(the defend-path analogue of a `player_id` spoof), the server currently emits
**no OTEL watcher event**. The GM panel — the lie-detector — can therefore not
see the rejection. The existing `player_id` spoof-rejection path already emits a
watcher event; the defend-path rejection must reach parity with it.

Per project doctrine (CLAUDE.md OTEL Observability Principle): every backend
subsystem decision must emit OTEL so the GM panel can verify the engine is
engaged rather than improvising. A silent authorization rejection violates that.

## Technical Approach
_Pointers for Dev (HOW is Dev's call — this is the where/what, not the code):_
- **Parity template:** locate the existing `player_id` spoof / seat-auth
  rejection OTEL watcher emit in `sidequest-server` (search the telemetry/watcher
  emit sites around player_id authorization / seat-auth). Mirror its event shape.
- **Emit site:** the defend-path authorization rejection from 126-8 — the
  seat/authorization check on the Fate DEFEND barrier. Add the matching watcher
  event at that rejection branch.
- Keep the new event's fields in parity with the spoof-rejection template
  (e.g. session/seat/actor + reason) so the GM panel renders both the same way.

## Scope
- In scope: emit one OTEL watcher event at the defend-path auth-rejection branch,
  with fields in parity with the existing `player_id` spoof-rejection event.
- Out of scope: changing the authorization decision itself, the defend mechanics,
  or any UI. This is observability only.
