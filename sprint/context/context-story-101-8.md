## Problem
Unify the three divergent slug-derivation rules behind one shared function.

PROBLEM (evidence-backed). Three different slug rules exist in the tree and DISAGREE on non-ASCII (diacritic) input:
  1. scripts/render_common.py:301 slugify — lowercases, strips a fixed punctuation set, spaces->'_', but KEEPS non-ASCII letters.
     "Srárný Fyzioloniązka" -> srárný_fyzioloniązka
  2. sidequest-server/sidequest/server/utils.py:4 slugify_player_name — lowercases, whitespace->'_', then DROPS every char not in [a-z0-9_-] (mirrors daemon sidequest_daemon.media.catalogs._slugify_name).
     "Srárný Fyzioloniązka" -> srrn_fyziolonizka
  3. sidequest-server/sidequest/server/reference_slug.py:18 slugify — replaces runs of [^a-z0-9]+ with a single '-', strips edges (non-ASCII becomes a hyphen separator).
     "Srárný Fyzioloniązka" -> sr-rn-fyzioloni-zka

CONSUMERS (blast radius):
  - Rule 2 (slugify_player_name) is used by reference_presenters.cast_portrait_slug -> the NPC PORTRAIT path. R2 portrait keys are generated by this same drop-non-ASCII rule, so the portrait path is INTERNALLY CONSISTENT.
  - Rule 3 (reference_slug.slugify) is used by reference_renderer.py, reference_anchors.py, reference_projection.py (map/POI/anchor slugs), and cli/validate/locations.py.
  - Rule 1 (render_common.slugify) is used by the render scripts.

WHY THIS MATTERS (and what it is NOT). This divergence is NOT the cause of the current evropi blank-portrait incident (that was a stale r2_manifest.json in the server's clone; being fixed separately). Do not conflate them.
The REAL exposure: the POI/map surface uses rule 3 (hyphenate) while assets may be rendered under rule 1 or rule 2 — so any non-ASCII POI / location name can mis-key its asset (page requests sr-rn-... while the file is srrn_... or srárný_...). Diacritic-heavy worlds (evropi, coyote_star) are exposed. This is a silent 404 / not_found class of bug.
