## Problem
The "is this a test/tool-test session?" predicate is encoded **twice, independently**:
- **Server:** `is_test_session` in `sidequest-server/sidequest/telemetry/watcher_hub.py:436` (used at :745)
- **UI:** `isTestSession` in `sidequest-ui/src/components/Dashboard/source/useLiveSource.ts:477` (used at :338, :364)

Each hard-codes the set of session-id prefixes (test-/tool-test-) that exclude a run from the
live GM dashboard. The two lists can drift silently, re-opening the 126-34 leak (test/tool runs
reappearing on the live dashboard). 126-34 fixed the filtering; this story keeps the two halves
in sync so the fix can't quietly regress.

## Technical Approach
Make the two predicates un-driftable. Either is acceptable — Dev picks the lightest:
1. **Cross-repo contract test:** a shared fixture set of session-ids with expected classifications;
   both server and ui tests assert their predicate matches the fixture, so a prefix added on one
   side without the other fails CI.
2. **Shared source of truth:** the prefix set lives in one place (e.g. a server-emitted constant /
   protocol field the ui consumes), eliminating the second hand-maintained list entirely.

Prefer whichever makes drift a **test failure, not a silent regression**. Do not change *what*
counts as a test session — this is a sync/guard, not a redefinition.

## Scope
- **In scope:** a contract test or shared source binding the two predicates; the canonical prefix
  fixture/constant; tests in both server and ui (or one cross-repo contract) that fail on drift.
- **Out of scope:** changing the predicate's behavior or the prefix semantics; unrelated dashboard
  filtering changes.

## Acceptance Criteria
- AC1: A single canonical definition of the test-session prefixes exists (shared constant or shared
  fixture), with both `is_test_session` (server) and `isTestSession` (ui) bound to it.
- AC2: A test fails if the server and ui predicates classify the same session-id differently
  (i.e. adding/removing a prefix on one side without the other breaks CI).
- AC3: Existing 126-34 behavior is unchanged — current test/tool-test sessions are still excluded
  from the live dashboard; non-test sessions still pass through.
- AC4: Both repos' existing test suites stay green.
