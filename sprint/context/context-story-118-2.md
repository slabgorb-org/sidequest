## Technical Approach
This is a **read-only display surface** — it consumes the `FATE_STATE` projection
already produced server-side in 118-1 (F3a, server #880: `build_fate_state_payload`
+ `_maybe_emit_fate_state`, gated on `ruleset == 'fate'`, emitted as the raw
display-only sibling of the sanitized compact projection). No new server work.

Mirror the existing reactive-panel pattern (do not invent a new one):
1. **`FatePanel.tsx`** — a new dockview panel component, structured like the existing
   `RelationshipsPanel.tsx` / `QuestsPanel.tsx` (same dockview registration, same
   genre-theme CSS-custom-prop approach, same null/empty handling). Renders, per PC:
   - fate-point count (current; labeled)
   - aspects grouped by kind: high-concept / trouble / character / situation /
     boost / consequence, each with free-invoke pips
   - skills on the Fate ladder: Terrible -2 … Legendary +8, **both** the rung label
     and the numeric value (this is the Sebastien/Jade legibility requirement)
   - consequence slots: mild / moderate / severe / extreme, rendered filled vs open
2. **`fateState` slice** — wire into `useStateMirror` + `GameStateProvider` following
   the existing relationships/quests slice pattern: idempotent full-replace on each
   `FATE_STATE` message, null-guard, and a **No-Silent-Fallback** log on a malformed
   payload (do not silently drop — log loudly per project doctrine).
3. **GameBoard registration** — register `FatePanel` in `GameBoard` alongside the
   other dockview panels.
4. **Genre theming** — drive colors/spacing from genre CSS custom properties, as the
   sibling panels do.

Doctrine guardrails for TEA/Dev:
- **No-Silent-Fallback:** malformed/absent `FATE_STATE` must log, not vanish.
- **Every-Test-Suite-Needs-a-Wiring-Test:** include an integration test proving the
  panel is registered in GameBoard and the slice is fed by the real `useStateMirror`
  message path — not just component-in-isolation tests.
- **Player-facing legibility:** every number must carry a label (fate points, ladder
  values, pip counts, consequence severities). This panel exists *for* the
  mechanics-first players.
- Read-surface only: no fate-point spending / aspect-invoke actions in this story.

## Scope
- **In scope:** `FatePanel.tsx`, the `fateState` mirror slice + provider wiring,
  GameBoard registration, genre-themed CSS, and tests (incl. a wiring test).
- **Out of scope:** any *interactive* fate actions (spending points, invoking
  aspects), server changes (FATE_STATE already ships from 118-1), and non-Fate
  rulesets (panel only renders when fate state is present).

## Acceptance Criteria
(TEA to formalize as failing tests in RED; derived from the story spec.)
1. `FatePanel` renders per-PC fate-point count with a visible label.
2. Aspects render grouped by all six kinds, each showing free-invoke pips.
3. Skills render on the Fate ladder with both rung label and numeric value
   (Terrible -2 … Legendary +8).
4. Consequence slots render for mild/moderate/severe/extreme, distinguishing
   filled vs open.
5. A `FATE_STATE` message updates the `fateState` slice via `useStateMirror`
   (idempotent full-replace); a malformed payload is logged, not silently dropped.
6. `FatePanel` is registered and reachable in `GameBoard` (wiring test).
7. Panel styling is driven by genre CSS custom properties.
